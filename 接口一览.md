# 接口一览（前端接入版）

说明
- 本清单聚焦“用户管理 + 成果管理”相关接口，按前端视角梳理：路径、方法、权限、字段说明（中英对照）、返回体与示例
- 与《实验室信息管理系统接口文档.md》保持一致，但更简明直观，便于前端查找与对接
- 鉴权：除公开接口外，均需在 Header 携带 Authorization: Bearer <token>

- 1. 用户（Users）

- 3. 我的成果（My Achievements）

## 1. 用户（Users）

备注：仅列出对成果管理有直接帮助的查询/检索接口；实际系统中还有登录、注册、权限等接口，此处从略。

1.1 模糊检索实验室用户（供前端选人用）
- GET /lab/users/search?keyword=
- 权限：已登录（建议；若使用 /lab/users/crud/search 则无需登录）
- 请求参数（Query）
  - keyword (string): 关键词，匹配 username/real_name/english_name/email/phone
- 返回：List<LabUserProfileDTO>（包含 email/phone/studentNumber，便于区分重名）

## 2. 成果管理（Achievements）

权限模型（简化说明）
- 管理员（Admin）: 可增删改查
- 拥有者（Owner）: 可编辑/删除自己的成果
- 第一作者（First Author, internal, author_order=1）: 可编辑/删除该成果

2.1 管理端列表（管理员/具备权限的用户）
- GET /lab/achievements?keyword=&type=&paperType=&projectType=&categoryId=&parentCategoryId=&published=&isVerified=&ownerUserId=&dateStart=&dateEnd=&pageNum=&pageSize=
- 权限：lab:achievement:list
- 说明：
  - keyword (string): 标题/关键词模糊; title/keywords
  - type (int): 1=论文(paper), 2=项目(project)
  - paperType (int): 1-7
  - projectType (int): 1-8
  - categoryId (long): 成果类型ID（新类型系统，推荐传二级分类ID，例如 4=会议论文, 15=青年基金）
  - parentCategoryId (long): 一级分类ID（聚合其所有二级分类一起分页）【1:论文，2:项目】
  - published (bool)
  - isVerified (bool)
  - ownerUserId (long)
  - dateStart/dateEnd (date): 论文匹配 publish_date，项目匹配 project_start_date
  - 兼容说明：当传入 parentCategoryId 或 categoryId 时，后端将忽略 type（type 为兼容参数）

2.2 详情
- GET /lab/achievements/{id}
- 权限：lab:achievement:query
- 返回：LabAchievementDTO（含作者列表）

2.3 新增
- POST /lab/achievements
- 权限：lab:achievement:add
- Body: CreateLabAchievementCommand
  - title (string) 标题
  - type (int) 1=paper, 2=project
  - paperType (int) 论文类型
  - projectType (int) 项目类型
  - categoryId (long) 成果类型ID（新类型系统，推荐传二级分类ID；更新接口同此）
  - publishDate (string yyyy) 论文发表年（type=1必填）
  - projectStartDate (string yyyy-MM) 项目开始（月）（type=2必填）
  - projectEndDate (string yyyy-MM 可空) 项目结束（月）

  - 类型自动补全与冲突规则（categoryId 优先）
    - 推荐：仅传“二级”categoryId（如 4=会议论文，15=青年基金）；后端自动补全旧字段：
      - 论文：自动补全 type=1、paperType=对应子类型
      - 项目：自动补全 type=2、projectType=对应子类型
    - 自定义“二级”categoryId（不在预置映射表内）将回退为：
      - 论文侧：paperType=7（其他论文）
      - 项目侧：projectType=8（其他项目）
    - 过渡期：写入时如仍传旧字段（type/paperType/projectType），后端将忽略并覆盖为 categoryId 推导值；后续将改为冲突即报错
    - 日期仍为必填：论文需 publishDate（yyyy）；项目需 projectStartDate（yyyy-MM；projectEndDate 如传必须不早于开始）

  - 示例：仅传“二级”categoryId 的论文
    ```json
    {
      "title": "测试-会议论文",
      "categoryId": 4,
      "publishDate": "2025",
      "published": true,

      "authors": []
    }
    ```
  - 示例：自定义“二级”categoryId=18（挂在“论文”顶级下），将自动补全为“其他论文”：paperType=7
    ```json
    {
      "title": "测试-自定义论文分类",
      "categoryId": 18,
      "publishDate": "2021",
      "published": true,
      "authors": []
    }
    ```

  - 更新接口同上述“categoryId 优先自动补全”规则；写入时如传旧字段，后端会忽略并覆盖为 categoryId 推导值（过渡期），后续将改为冲突即报错

  - venue/linkUrl/gitUrl/homepageUrl/pdfUrl/doi/coverUrl (string)
  - fundingAmount (decimal)
  - authors (array<CreateAuthorCommand>) 批量作者：
    - userId (long) 内部作者ID（强烈推荐）
    - username/email/phone/studentNumber（若未传 userId，用于自动绑定）
    - name/nameEn (string) 姓名（外部作者必填；自动绑定失败时保留为外部作者）
    - authorOrder (int>0) 作者顺序（唯一）
    - isCorresponding (bool)
    - role (string)
    - visible (bool) 仅对内部作者生效

2.4 修改
- PUT /lab/achievements/{id}
- 权限：lab:achievement:edit
- 可操作人：管理员/拥有者/第一作者（内部，author_order=1）
- Body：同 CreateLabAchievementCommand；若包含 authors，将全量替换：
  - 逻辑：硬删历史软删作者 -> 软删当前未删作者 -> 插入新列表

2.5 删除
- DELETE /lab/achievements/{id}
- 权限：lab:achievement:remove
- 可操作人：管理员/拥有者/第一作者（内部，author_order=1）
- 行为：成果软删，同时软删该成果下未删作者；作者端不可见

## 2.6 成果管理 v2（仅叶子 categoryId，拒收旧字段）
- 基路径：/v2/lab/achievements
- 写接口与权限：沿用 v1 相同权限标识（lab:achievement:add/edit/remove 等）
- 设计目标：写入端彻底去除 legacy 字段（type/paperType/projectType），仅提交二级分类 ID（categoryId）
- 兼容策略：服务端内部会按 categoryId 自动推导旧字段，读侧保持兼容

2.6.1 新增（v2）
- POST /v2/lab/achievements
- 权限：lab:achievement:add
- 请求体：CreateLabAchievementV2Command（仅列出与 v2 相关的关键字段）
  - title (string, required)：标题，<=500
  - categoryId (long, required)：必须为叶子（二级）分类 ID
  - venue (string, optional)：期刊/会议/发布渠道，<=300
  - publishDate (string, optional)：论文年份，格式 yyyy
  - projectStartDate/projectEndDate (string, optional)：项目起止时间，格式 yyyy-MM
  - linkUrl/gitUrl/homepageUrl/pdfUrl/doi (string, optional)：外链/仓库/主页/PDF/DOI
  - reference (string, optional)：参考文献，<=5000
  - fundingAmount (number, optional)：项目经费（万元），>=0
  - published (bool, optional)：是否对外发布，默认 false
  - authors (array<CreateAuthorCommand>, optional)：作者列表（语义与 v1 相同）
- 严格拒收字段：type、paperType、projectType 一旦出现在请求体中即 400
- 请求示例：
```json
{
  "title": "测试-会议论文",
  "categoryId": 24,
  "publishDate": "2025",
  "published": true,
  "authors": []
}
```
- 成功响应：
```json
{
  "code": 200,
  "msg": "操作成功",
  "data": 123  // 新建的成果ID
}
```
- 常见错误：
  - 400：{"code":103,"msg":"必须传二级分类ID: categoryId"}
  - 400：{"code":103,"msg":"v2 接口不允许提交 legacy 字段：type"}
  - 400：{"code":103,"msg":"请求参数异常，字段校验失败 …"}

2.6.2 修改（v2）
- PUT /v2/lab/achievements/{id}
- 权限：lab:achievement:edit
- 请求体同“新增（v2）”，同样拒收 type/paperType/projectType
- 成功响应：
```json
{ "code": 200, "msg": "操作成功", "data": null }
```
- 注意：若传 authors，将采用与 v1 一致的“全量替换”语义

2.6.3 业务说明
- 分类要求：categoryId 必须为叶子（二级）分类；顶级分类或不存在的 ID 将被拒绝
- 自动推导：后端根据 categoryId 自动判定大类（论文/项目）与子类型，内部回填 legacy 字段（对调用方透明）
- 规则校验：继续根据大类执行合理的业务校验（如论文类可能校验 publishDate，项目类可能校验起止时间）
- 读侧兼容：查询/详情仍可能返回 legacy 字段用于兼容老前端，后续会逐步收敛为仅展示 category 信息

2.6.4 迁移建议
- 前端：把 v1 写接口（POST/PUT /lab/achievements）全部切到 v2（/v2/lab/achievements）
  - 移除请求体中的 type/paperType/projectType，仅保留 categoryId 与其他字段
- 后端：保留 v1 读写一段时间用于过渡，但标记为“废弃”，后续计划移除 legacy 字段依赖


## 3. 我的成果（My Achievements）

3.1 列表（我拥有的 + 我参与的）
- GET /lab/my-achievements?keyword=&type=&paperType=&projectType=&categoryId=&parentCategoryId=&published=&isVerified=&ownerName=&authorName=&dateStart=&dateEnd=&pageNum=&pageSize=
- 权限：已登录
- 说明：
  - keyword：标题/关键词模糊
  - ownerName：拥有者姓名（模糊匹配 real_name/english_name；仅在“我的范围”内过滤）
  - authorName：作者姓名（模糊匹配作者表 name/name_en + 内部作者姓名；仅在“我的范围”内过滤）
  - categoryId：二级分类ID（叶子分类）
  - parentCategoryId：一级分类ID（聚合其所有二级分类一起分页）
  - 兼容说明：当传入 parentCategoryId 或 categoryId 时，后端将忽略 type（type 已标注为 Deprecated）
- 返回：PageDTO<LabAchievementDTO>（含全部作者，不过滤 visible）

3.1.1 字段表与示例
- 一级分类聚合分页
```
GET /lab/my-achievements?parentCategoryId=<一级ID>&pageNum=1&pageSize=20
```
- 二级分类分页
```
GET /lab/my-achievements?categoryId=<二级ID>&pageNum=1&pageSize=20
```

3.2 切换“我”在某成果中的个人页可见性（仅内部作者）
- PUT /lab/my-achievements/{achievementId}/visibility?visible=true|false
- 权限：已登录（仅作者本人能改自己）

## 4. 成果作者（Authors）

基础路径：/lab/achievements/{achievementId}/authors

4.1 列表
- GET /
- 权限：lab:achievement:query
- 返回：List<LabAchievementAuthorEntity>（按 author_order 排序）

4.2 新增作者（内部/外部作者均支持）
- POST /
- 权限：lab:achievement:edit（且需拥有者或管理员）
- Body: AddAuthorCommand
  - 入参优先级与自动绑定：
    - 第一优先：userId（直接绑定为内部作者；同一成果下同一 userId 仅一次）
    - 第二优先：username/email/phone/studentNumber（按此优先级唯一解析为 lab_user 并绑定）
    - 第三优先：name/nameEn（仅当唯一姓名命中时才自动绑定；否则作为外部作者）
  - 常见规则：
    - 作者顺序 authorOrder 必须唯一且 > 0
    - 若新增内部作者遇到数据库唯一冲突(achievement_id, user_id)，接口会：
      - 若存在且未删：报“该内部作者已存在于该成果”
      - 若存在但软删：直接“复活”并更新该作者信息

4.3 更新作者
- PUT /{authorId}
- 权限：lab:achievement:edit（且需拥有者或管理员）
- Body: UpdateAuthorCommand（可更新 userId/name/nameEn/affiliation/authorOrder/isCorresponding/role/visible）

4.4 删除作者
- DELETE /{authorId}
- 权限：lab:achievement:edit（且需拥有者或管理员）
- 行为：软删作者

4.5 调整作者顺序
- PUT /{authorId}/reorder?newOrder=1..n
- 权限：lab:achievement:edit（且需拥有者或管理员）
- 规则：作者顺序唯一

4.6 切换作者可见性（仅内部作者）
- PUT /{authorId}/visibility?visible=true|false
- 权限：lab:achievement:edit（且需拥有者或管理员）

5.4 创建类型
- POST /lab/achievement-categories
- 权限：`lab:category:add`
- Body: CreateCategoryCommand
  - parentId (long|null): 父类型ID，null 表示一级类型
  - categoryCode (string): 类型编码（唯一，可选；未提供则后端按名称自动生成）
  - categoryName (string, 必填): 类型名称
  - categoryNameEn (string): 英文名称
  - description (string): 描述
  - sortOrder (int): 排序（可选）
  - isActive (bool): 是否启用，默认 true
  - icon (string): 图标
  - color (string): 颜色标识（如 #1890ff）
- 返回：Long（新建类型ID）
- 请求示例（一级类型）
```bash
curl -X POST "http://localhost:8080/lab/achievement-categories" \
 -H "Authorization: Bearer <token>" -H "Content-Type: application/json" \
 -d '{
   "parentId": null,
   "categoryName": "自定义一级",
   "categoryNameEn": "Custom Top",
   "description": "说明",
   "sortOrder": 99,
   "isActive": true,
   "icon": "icon-name",
   "color": "#1890ff"
 }'
```
- 请求示例（二级类型）
```bash
curl -X POST "http://localhost:8080/lab/achievement-categories" \
 -H "Authorization: Bearer <token>" -H "Content-Type: application/json" \
 -d '{
   "parentId": 1,
   "categoryCode": "CUSTOM_SUB",
   "categoryName": "自定义二级"
 }'
```
- 返回字段示例（ResponseDTO 包装）
```json
{
  "code": 0,
  "msg": "操作成功",
  "data": 123
}
```

5.8 删除类型
- DELETE /lab/achievement-categories/{id}
- 权限：`lab:category:remove`
- 行为：软删除
- 限制：系统内置类型不可删除；存在子类型或已被成果使用时不可删除
- 请求示例：
```bash
curl -X DELETE "http://localhost:8080/lab/achievement-categories/12" \
 -H "Authorization: Bearer <token>"
```
- 返回示例：
```json
{ "code": 0, "msg": "操作成功", "data": null }
```
- 失败返回示例：
```json
{ "code": 12801, "msg": "该类型存在成果，不允许删除", "data": null }
```
```json
{ "code": 12802, "msg": "该类型存在下级类型，不允许删除", "data": null }
```


5.9 前端对接要点
- 一级类型：parentId=null；二级类型：parentId=对应顶级ID

### 4.A 作者管理对接指南

1) 新增成果时如何新增作者
- 推荐：在“创建成果”的同一个请求里携带 authors 数组，一次性创建成果及作者
- 接口：POST /lab/achievements
- Body（CreateLabAchievementCommand）关键字段：
  - authors: 数组<CreateAuthorCommand>
    - userId (number) 内部作者ID（强烈推荐）
    - username/email/phone/studentNumber（未传 userId 时用于自动绑定）
    - name/nameEn (string) 姓名（外部作者必填；自动绑定失败则作为外部作者保存）
    - authorOrder (number>0) 作者顺序（同一成果下唯一）
    - isCorresponding (boolean)
    - role (string)
    - visible (boolean) 仅对内部作者生效
- 示例：
  {
    "title": "test111",
    "type": 1,
    "paperType": 1,
    "publishDate": "2025",
    "venue": "",
    "authors": [
      { "userId": 28, "name": "袁鹏辉", "authorOrder": 1, "visible": true },
      { "name": "佟强", "authorOrder": 2, "visible": true }
    ]
  }
- 规则补充：
  - 自动绑定优先级：userId > username/email/phone/studentNumber > 唯一中文名/英文名
  - 同一(achievement_id, user_id) 若历史软删存在 → 复活并更新；未删已存在 → 返回业务错误“该内部作者已存在于该成果”；不存在 → 插入

2) 修改成果时如何对作者增/改/删
- 方案A（推荐，表单一次性提交）：全量替换作者列表
  - 接口：PUT /lab/achievements/{id}
  - Body：与创建一致，authors 传“替换后”的完整列表
  - 后端行为：清理历史软删 → 软删当前未删 → 逐条写入（内部作者 upsert；外部作者插入）
- 方案B（细粒度逐项操作）：
  - 新增作者：POST /lab/achievements/{achievementId}/authors
  - 更新作者：PUT /lab/achievements/{achievementId}/authors/{authorId}
  - 删除作者：DELETE /lab/achievements/{achievementId}/authors/{authorId}
  - 调整顺序：PUT /lab/achievements/{achievementId}/authors/{authorId}/reorder?newOrder=2
- 权限：作者管理（增/改/删/调序）需“拥有者或管理员”

3) 删除成果时作者表会怎样
- 接口：DELETE /lab/achievements/{id}
- 可操作人：管理员 或 拥有者 或 第一作者（内部且 author_order=1）
- 行为：成果软删；作者清理=先硬删历史软删作者，再软删当前未删作者；作者端不可见

前端建议
- 新增/更新作者优先让用户选择实验室用户（search-select 拿 userId），确保为内部作者（会出现在“我的成果”）
- 仅姓名录入时提示：仅在“唯一命中”时才会自动绑定为内部作者，重名将保存为外部作者
- 在前端保证 authorOrder 唯一，避免后端报错

## 7. 公开接口（对外展示，无需登录）

7.1 公开成果列表
- GET /open/achievements?keyword=&type=&paperType=&projectType=&categoryId=&parentCategoryId=&dateStart=&dateEnd=&authorName=&pageNum=&pageSize=
- 权限：公开（无需登录）
- 说明：返回所有未删除的成果，支持多种筛选条件
- 请求参数（Query，均可选）：
  - keyword (string): 关键词，匹配 title/keywords
  - type (int)【Deprecated/兼容参数】：1=论文，2=项目；当传入 categoryId 或 parentCategoryId 时，后端将忽略 type
  - paperType (int): 论文类型（1-7），仅type=1时生效
  - projectType (int): 项目类型（1-8），仅type=2时生效
  - categoryId (long): 二级分类ID（叶子分类）；按该分类筛选
  - parentCategoryId (long): 一级分类ID；聚合其所有二级分类下的成果进行分页（传此参数时无需再传 categoryId）

  - dateStart (date): 开始日期，格式：yyyy-MM-dd
  - dateEnd (date): 结束日期，格式：yyyy-MM-dd
  - authorName (string): 作者姓名（模糊搜索，匹配中文名/英文名）
  - pageNum (int): 页码，默认1
  - authorUserId (long): 内部作者用户ID（精确匹配，推荐在“个人页”使用）

  - pageSize (int): 每页大小，默认10
- 返回：PageDTO<PublicAchievementDTO>
- 示例：
  ```
  GET /open/achievements?authorName=张三&type=1&pageNum=1&pageSize=10
  ```

### 数据过滤规则
- **成果过滤**：仅返回未删除的成果（deleted=false），不限制发布状态

- **用户过滤**：仅返回激活状态的用户（is_active=true, deleted=false）

- `lab:user:query`: 查看用户详情

- `lab:user:edit`: 编辑用户

- `@labUserPermission.canViewUser(#userId)`: 检查是否可以查看指定用户
- `@labUserPermission.canViewUserByUsername(#username)`: 检查是否可以查看指定用户名的用户

### 8.6 业务规则

#### 用户创建规则
1. **用户名唯一性**: 系统内用户名必须唯一
2. **学号唯一性**: 学号不能重复（可为空）
3. **手机号唯一性**: 手机号不能重复（可为空）
4. **邮箱唯一性**: 邮箱不能重复（可为空）
5. **默认密码**: 创建用户时密码为6-20位字符

#### 权限控制规则
1. **管理员权限**: 可以管理所有用户
2. **个人信息**: 用户只能查看和修改自己的信息
3. **密码安全**: 修改密码需要验证旧密码
4. **账号状态**: 禁用的账号无法登录

#### 数据完整性规则
1. **软删除**: 删除用户时保留历史数据
2. **关联数据**: 删除用户前需要处理关联的成果数据
3. **状态管理**: 毕业/离职用户保留数据但标记状态

#### 文件上传规则
1. **头像上传**: 支持jpg、jpeg、png、gif、bmp格式
2. **文件大小**: 最大50MB
3. **绝对存储路径**: `/tmp/agileboot/profile/avatar/` (开发环境)
4. **访问URL**: `http://localhost:8080/profile/avatar/文件名`
5. **数据库存储**: 相对路径格式 `/profile/avatar/文件名`
6. **权限控制**: 用户只能上传自己的头像，管理员可以为任何用户上传
7. **文件命名**: 自动生成唯一文件名，格式：`时间戳_原文件名_UUID.扩展名`
8. **路径配置**: 通过 `agileboot.file-base-dir` 配置基础路径，生产环境需要修改

## 9. 师生关系管理（Teacher-Student Relationship）

### 9.1 师生关系CRUD管理

9.1.1 创建师生关系
- **接口地址**: `POST /lab/users/relationships`
- **权限**: 管理员 (`lab:user:relationship:create`)
- **说明**: 建立导师与学生的指导关系
- **请求体**: CreateUserRelationshipCommand
```json
{
  "teacherId": 38,
  "studentId": 29,
  "relationshipType": 1,
  "startDate": "2025-09-11",
  "remark": "测试师生关系-刘秀磊导师指导李金峰"
}
```
- **返回**: Long（关系ID）
- **业务规则**：
  - 只有身份为"教师"的用户才能作为导师
  - 只有身份为"学生"的用户才能作为学生
  - 一个学生同时只能有一个活跃的导师
  - 一个导师可以有多个学生
  - 不能建立自己与自己的师生关系

9.1.2 解除师生关系（硬删除）
- **接口地址**: `DELETE /lab/users/relationships/teacher/{teacherId}/student/{studentId}`
- **权限**: 管理员 (`lab:user:relationship:delete`)
- **说明**: 解除导师与学生的指导关系（永久删除记录）
- **路径参数**：
  - `teacherId` (long): 导师ID
  - `studentId` (long): 学生ID
- **返回**: Void
- **注意**: ⚠️ 此操作为硬删除，记录将被永久删除且无法恢复

9.1.3 获取导师的学生列表
- **接口地址**: `GET /lab/users/relationships/teacher/{teacherId}/students`
- **权限**: 管理员或导师本人 (`lab:user:relationship:query`)
- **说明**: 获取指定导师的所有学生
- **路径参数**: `teacherId` (long)
- **查询参数**: `status` (int, 可选) - 关系状态：1=活跃,2=已结束，不传查询所有
- **返回**: List<UserRelationshipDTO>

9.1.4 获取学生的导师信息
- **接口地址**: `GET /lab/users/relationships/student/{studentId}/supervisor`
- **权限**: 管理员或学生本人 (`lab:user:relationship:query`)
- **说明**: 获取指定学生的导师信息
- **路径参数**: `studentId` (long)
- **查询参数**: `status` (int, 可选) - 关系状态：1=活跃,2=已结束，不传查询所有
- **返回**: UserRelationshipDTO

9.1.5 更换学生的导师
- **接口地址**: `PUT /lab/users/relationships/student/{studentId}/supervisor/{newTeacherId}`
- **权限**: 管理员 (`lab:user:relationship:update`)
- **说明**: 更换学生的导师（结束旧关系，创建新关系）
- **路径参数**：
  - `studentId` (long): 学生ID
  - `newTeacherId` (long): 新导师ID
- **查询参数**: `remark` (string, 可选) - 备注信息
- **返回**: Long（新关系ID）

9.1.6 为导师添加学生（快捷方式）
- **接口地址**: `POST /lab/users/relationships/teacher/{teacherId}/student/{studentId}`
- **权限**: 管理员 (`lab:user:relationship:create`)
- **说明**: 快捷方式为导师添加学生
- **路径参数**：
  - `teacherId` (long): 导师ID
  - `studentId` (long): 学生ID
- **查询参数**: `remark` (string, 可选) - 备注信息
- **返回**: Long（关系ID）

9.1.7 获取活跃的导师学生列表
- **接口地址**: `GET /lab/users/relationships/teacher/{teacherId}/active-students`
- **权限**: 管理员或导师本人 (`lab:user:relationship:query`)
- **说明**: 获取指定导师的活跃学生列表
- **路径参数**: `teacherId` (long)
- **返回**: List<UserRelationshipDTO>

9.1.8 获取学生的活跃导师
- **接口地址**: `GET /lab/users/relationships/student/{studentId}/active-supervisor`
- **权限**: 管理员或学生本人 (`lab:user:relationship:query`)
- **说明**: 获取指定学生的活跃导师信息
- **路径参数**: `studentId` (long)
- **返回**: UserRelationshipDTO

### 9.6 业务规则

#### 关系建立规则
1. **身份验证**: 导师必须是教师身份（identity=2），学生必须是学生身份（identity=3）
2. **唯一性约束**: 一个学生同时只能有一个活跃的导师
3. **自引用检查**: 不能建立自己与自己的师生关系
4. **状态管理**: 新建关系默认为活跃状态（status=1）

#### 关系管理规则
1. **硬删除**: ⚠️ 解除关系时使用硬删除，记录将被永久删除且无法恢复
2. **数据一致性**: 删除后所有相关查询都会正确更新
3. **级联查询**: 查询时自动关联用户基本信息
4. **权限隔离**: 用户只能查看与自己相关的师生关系

#### 重要注意事项
- ⚠️ **硬删除风险**: 删除操作不可逆，请谨慎操作
- 📊 **数据完整性**: 删除前建议备份重要数据
- 🔒 **权限控制**: 只有管理员可以执行删除操作

### 9.7 数据库设计

#### lab_user_relationship 表结构（硬删除版本）
```sql
CREATE TABLE lab_user_relationship (
  id BIGINT PRIMARY KEY AUTO_INCREMENT COMMENT '关系ID',
  teacher_id BIGINT NOT NULL COMMENT '导师ID',
  student_id BIGINT NOT NULL COMMENT '学生ID',
  relationship_type TINYINT DEFAULT 1 COMMENT '关系类型：1=导师关系',
  start_date DATE COMMENT '关系开始时间',
  end_date DATE COMMENT '关系结束时间',
  status TINYINT DEFAULT 1 COMMENT '状态：1=活跃,2=已结束',
  remark VARCHAR(255) COMMENT '备注信息',

  -- 审计字段
  creator_id BIGINT COMMENT '创建者ID',
  create_time DATETIME DEFAULT CURRENT_TIMESTAMP COMMENT '创建时间',
  updater_id BIGINT COMMENT '更新者ID',
  update_time DATETIME DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP COMMENT '更新时间',

  -- 索引和约束
  UNIQUE KEY uk_teacher_student (teacher_id, student_id) COMMENT '导师学生唯一约束',
  KEY idx_teacher (teacher_id) COMMENT '导师索引',
  KEY idx_student (student_id) COMMENT '学生索引',
  KEY idx_status (status) COMMENT '状态索引',
  KEY idx_teacher_status (teacher_id, status) COMMENT '导师状态复合索引',
  KEY idx_student_status (student_id, status) COMMENT '学生状态复合索引',
  KEY idx_active_relationships (status, teacher_id, student_id) COMMENT '活跃关系复合索引',
  KEY idx_create_time (create_time) COMMENT '创建时间索引',

  -- 外键约束
  CONSTRAINT fk_teacher FOREIGN KEY (teacher_id) REFERENCES lab_user(id) ON DELETE CASCADE,
  CONSTRAINT fk_student FOREIGN KEY (student_id) REFERENCES lab_user(id) ON DELETE CASCADE
) COMMENT='师生关系表（硬删除）';
```

#### 与软删除版本的区别
1. **移除deleted字段**: 不再使用软删除标记
2. **简化唯一约束**: `uk_teacher_student` 不再包含deleted字段
3. **优化索引结构**: 移除deleted相关的复合索引
4. **级联删除**: 用户删除时自动清理关系记录
